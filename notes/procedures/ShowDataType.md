# DataType — Show

Read a single data type by full path and return its full description (struct
fields, enum entries, typedef base, pointer/array base, etc.) plus a C
declaration. Read-only.

## Request
```typescript
interface ShowDataTypeRequest {
  procedure: "ShowDataType";
  file: string;
  path: string;            // full type path, e.g. "/ELF/Elf64_Ehdr" or "/int"
}
```

## Response

```typescript
interface ShowDataTypeResponse {
  success: true;
  kind: "struct" | "union" | "enum" | "typedef"
       | "pointer" | "array" | "functiondef" | "bitfield" | "primitive";
  name: string;
  path: string;
  category: string;
  size: number;            // byte length; -1 for variable-length primitives
  source: "USER" | "BUILTIN" | "ARCHIVE";
  sourceArchive: string | null;

  // Full structured view (same shape as the pre-C output). Always present.
  detail: {
    fields?: FieldEntry[];       // struct/union
    entries?: EnumEntry[];       // enum
    base?: string;               // typedef / pointer / array base (display name)
    count?: number;              // array element count
    signature?: string;          // function definition prototype
    bitOffset?: number;          // bitfield
    bitSize?: number;
    packed?: boolean;            // struct
    alignment?: number;          // struct
    description?: string;
    // Plus the top-level scalar fields above (kind, name, path, ...) mirrored
    // inside the object for callers that just want to feed `detail` to a JSON
    // consumer without knowing the wrapper shape.
  };

  // C declaration, generated by Ghidra's DataTypeWriter (the same engine the
  // GUI's "Export C" uses). Always present; may be empty for kinds
  // DataTypeWriter cannot format (CLI falls back to the JSON `detail` in
  // that case). The CLI prints this by default and prints `detail` when
  // `--json` is given.
  c: string;
}
interface FieldEntry {
  name: string;                // "" for unnamed fields
  offset: number;
  size: number;
  type: string;                // C-syntax display name, e.g. "char *", "MyStruct[4]"
}
interface EnumEntry {
  name: string;
  value: number;               // long, but JS numbers are fine
}
```

## CLI

```
ghidra-headless-cli datatype show --file /Mapeditor.exe --path /MyStruct
```

prints the C declaration by default. Example output:

```
struct   /MyStruct   32
typedef unsigned char    byte;
typedef unsigned int    dword;
...
typedef struct MyStruct MyStruct, *PMyStruct;

struct MyStruct {
    int magic;
    longlong id;
    char name[16];
    void *next;
};
```

`--json` switches to the raw structured view:

```
ghidra-headless-cli datatype show --file /Mapeditor.exe --path /MyStruct --json
struct   /MyStruct   32
kind: struct
name: MyStruct
path: /MyStruct
category: /
size: 32
source: ARCHIVE
sourceArchive: Mapeditor.exe
fields: [{"name":"magic","offset":0,"size":4,"type":"int"}, ...]
packed: true
alignment: 8
```

The headline TSV line `kind<TAB>path<TAB>size` is always emitted first so
scripts that pipe to `head -1` or awk on the path still work in either mode
(it's the one piece of output identical between the two).

## Notes

- `path` resolution: the last `/` separates the category path from the type
  name. `/byte` is the root category + name "byte". `ELF/Elf64_Ehdr` and
  `/ELF/Elf64_Ehdr` both resolve to category `/ELF` + name `Elf64_Ehdr`.
- Field types in `detail.fields[].type` render as **C-syntax display names**
  (`char *`, `MyStruct[4]`) to avoid infinite recursion through nested
  struct types. Use `ShowDataType` recursively if you need the full
  field-detail of a nested type.
- The `c` field is what Ghidra's `DataTypeWriter.write(DataType[], ...)`
  produces. It includes referenced typedefs (so a struct referencing
  `longlong` pulls in `typedef long long longlong;`), a forward typedef
  for the type itself (`typedef struct MyStruct MyStruct, *PMyStruct;`),
  and the body.
- **No C → error, not fallback.** If the server's `DataTypeWriter` throws
  (rare; `IOException`/`CancelledException`) the server returns an empty
  `c` string. The CLI treats this as a hard error: it logs a clear
  message naming the type, path, and kind, and exits non-zero. It does
  NOT silently fall back to the JSON `detail` — the user asked for C,
  re-routing output without telling them would be worse than failing.
  The error message tells the user to retry with `--json` to see the
  structured view.
- `size: -1` means the type is variable-length (e.g. `TerminatedCString`).
- `category` on built-in primitives is always `"/"` (the root category).
- The same response envelope is used by `CreateDataType`, `ReplaceDataType`,
  and `EditDataType` for their confirmation output (so `datatype create`
  prints the same C declaration by default, which is the right "I just
  created a struct, show me what it looks like" experience).
